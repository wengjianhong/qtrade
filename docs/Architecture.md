# 量化交易系统架构设计

## 一、系统定位与目标

### 1.1 系统定位

本系统定位为**面向机构客户的企业级多租户量化交易平台**，支撑从策略研发、回测、仿真交易到实盘交易的全生命周期管理，适配高频 / 中高频 / 低频等多类交易场景，满足金融机构合规、高可用、高性能、可扩展、可审计的核心诉求，支持多账户、多交易所、多策略并行运行，具备跨地域部署与容灾能力。

### 1.2 核心目标

- **极致性能**：核心交易链路平均延迟 < 50 微秒，峰值订单处理能力 ≥ 10 万笔 / 秒
- **金融级高可用**：核心交易层全年可用性 ≥ 99.99%，支撑跨机房灾备，故障恢复时间（RTO）< 30 秒，数据恢复点目标（RPO）< 1 分钟
- **多租户隔离**：支持机构内多部门 / 多团队隔离，资源（CPU / 内存 / 行情 / 订单通道）按需分配，数据与操作权限严格隔离
- **合规可控**：满足金融监管要求（如交易留痕、反洗钱、异常交易监控），支持等保三级、数据脱敏与跨境数据合规
- **全生命周期可观测**：覆盖指标、日志、链路、调用链全维度监控，支持根因分析与性能溯源
- **弹性扩展**：核心层支持水平扩展，支撑服务层支持秒级弹性伸缩，插件化架构适配业务快速迭代



## 二、系统架构总览

### 架构层次：

本系统采用 “**核心模块进程内模块化集成 + 支撑功能微服务化 + 可插拔实现 + 企业级治理与容灾**” 的混合架构，平衡低延迟交易需求与企业级系统的稳定性、扩展性、合规性。

整体分为四层：

- **核心交易层**：进程内模块化设计，处理实时交易链路（微秒级响应），补充故障隔离、多核优化、本地容灾能力
- **支撑服务层**：独立部署的微服务集群，提供非实时性支撑功能，强化服务治理、熔断降级、弹性伸缩能力
- **接入层**：负责外部系统对接与用户交互，补充多租户、统一认证、安全网关能力
- **企业级治理层**：跨层治理能力，覆盖服务注册发现、配置灰度、容灾备份、合规审计、可观测性等企业级能力

### 核心设计原则：

- **低延迟优化**：核心交易链路全程内存交互，无网络开销，CPU 亲和性绑定，内存锁优化，网卡中断隔离
- **高可用性**：核心模块多实例部署 + 跨机房灾备，微服务集群化 + 熔断降级，关键组件主备切换 + 故障自动转移
- **可扩展性**：通过插件机制扩展数据源 / 策略 / 执行渠道，无需修改核心代码；支撑服务层支持水平扩容，核心层支持多进程分片部署
- **安全性**：多层风控（事前 / 事中 / 事后），操作审计，数据加密传输，多租户权限隔离，等保三级合规，敏感数据脱敏
- **可维护性**：模块化边界清晰，微服务独立升级，插件化降低耦合，全维度可观测，自动化运维（发布 / 回滚 / 监控）
- **多租户隔离**：资源隔离、数据隔离、权限隔离，租户级配置与策略独立管理
- **合规性**：全流程交易留痕，审计日志不可篡改，满足金融监管与行业合规要求

## 七、可插拔设计详细说明

### 1. 行情数据源可插拔设计

- **抽象接口层**：定义`IDataSource`标准接口，包含：

  - 连接管理：`Connect(config)`、`Disconnect()`
  - 订阅控制：`Subscribe(instruments)`、`Unsubscribe(instruments)`
  - 数据回调：`OnTick(tick_data)`、`OnBar(bar_data)`
  - 状态查询：`IsConnected()`、`GetSupportedInstruments()`
  - 企业级扩展：`GetHealthStatus()`（健康状态上报）、`GetErrorCode()`（标准化错误码）、`SetPriority()`（数据源优先级）

- **插件实现层**：

  - 每种数据源（如 CTP、Binance、Wind）实现`IDataSource`接口
  - 编译为独立动态库（.so/.dll），包含插件元数据（支持的品种、数据类型、版本、兼容系统版本）
  - 实现数据协议解析与转换（统一为系统内部格式）
  - 企业级扩展：1. 插件签名校验，防止恶意插件；2. 插件版本兼容检测，避免版本冲突；3. 插件运行状态监控，上报资源占用 / 错误率

- **管理组件**：

  - 数据源管理器：负责插件加载、配置解析、生命周期管理
  - 数据融合器：多数据源时按优先级合并数据（如主备数据源切换）
  - 插件注册中心：维护可用数据源插件列表与版本信息
  - 企业级扩展：1. 插件沙箱，隔离插件运行环境；2. 插件灰度加载，支持部分实例升级；3. 插件故障自动卸载，降级至备用插件；4. 插件操作审计，记录加载 / 卸载 / 切换日志

  

### 2. 策略实现可插拔设计

- **抽象基类层**：定义`StrategyBase`基类，包含：

  - 生命周期方法：`Init(config)`、`Start()`、`Pause()`、`Resume()`、`Stop()`
  - 事件处理方法：`OnTick(tick)`、`OnBar(bar)`、`OnOrder(order)`、`OnTrade(trade)`
  - 交互方法：`SendSignal(signal)`、`GetParameter(key)`、`SetParameter(key, value)`
  - 企业级扩展：`GetResourceUsage()`（资源占用上报）、`SetResourceLimit()`（资源限制）、`OnRiskAlert()`（风控告警回调）、`GetVersion()`（策略版本）

- **策略实现层**：

  - 具体策略继承`StrategyBase`，实现业务逻辑
  - 编译为策略插件（.so/.dll），包含策略元数据（名称、作者、参数描述、版本、风险等级）
  - 支持自定义指标计算与信号生成逻辑
  - 企业级扩展：1. 策略签名校验，防止篡改；2. 策略运行沙箱，隔离内存 / CPU 资源；3. 策略异常自动恢复，支持快照重启；4. 策略参数加密存储，防止泄露

- **管理组件**：

  - 策略工厂：根据配置动态加载策略插件，创建实例
  - 策略调度器：管理多策略并行运行，隔离资源
  - 策略仓库：存储策略插件，支持版本管理与灰度发布
  - 企业级扩展：1. 策略灰度发布管理器（按账户 / 品种 / 时间放量）；2. 策略资源调度器（按优先级分配 CPU / 内存）；3. 策略故障隔离器（单个策略崩溃不影响其他策略）；4. 策略审计日志（加载 / 启动 / 停止 / 参数修改全记录）

  

### 3. 下单执行模块可插拔设计

- **抽象接口层**：定义`IExecutionAdapter`接口，包含：

## 八、模块与微服务交互方式（企业级可靠性增强）

### 8.1 交互方式说明

- **核心模块内部**：通过内存事件总线（无锁队列）传递事件，同步 / 异步处理；补充事件优先级、事件持久化、故障恢复机制

- **核心层→微服务**：

  1. 同步调用：通过 gRPC 调用（带超时 / 重试 / 熔断），支持负载均衡与故障转移；
  2. 异步通知：通过消息队列（Kafka/RabbitMQ）异步通知，消息持久化 + 确认机制，确保至少一次投递；
  3. 企业级扩展：调用链追踪（OpenTelemetry），记录调用耗时 / 成功率，支持根因分析

- **微服务→核心层**：

  1. 通过消息队列推送事件（如配置更新），核心层监听指定主题，消息按租户分片；
  2. 核心层提供回调接口，微服务调用前需认证，防止未授权访问；
  3. 企业级扩展：消息幂等处理，防止重复触发核心逻辑

- **微服务之间**：

  1. 通过 gRPC 或消息队列交互，支持服务注册发现与负载均衡；
  2. 补充熔断降级、超时重试机制，防止服务雪崩；
  3. 跨机房微服务调用优先选择同机房实例，降低延迟；
  4. 调用链全链路追踪，支持微服务间性能分析

  

### 8.2 接口协议与交互规范

  #### 协议选型：HTTP vs gRPC（分场景决策）

结合系统低延迟核心诉求与多模块交互特性，采用分层协议策略，而非单一协议选型：

| 交互场景                               | 推荐协议 | 核心理由                                                     |
| :------------------------------------- | :------- | :----------------------------------------------------------- |
| 核心交易层 ↔ 支撑服务层                | gRPC     | 1. Protobuf 二进制序列化比 HTTP+JSON 快数倍，HTTP/2 多路复用减少连接开销，降低毫秒级延迟；<br>2. C++20 对 gRPC 生态支持完善（协程、强类型），契合核心层开发栈；<br/>3. 原生支持流式 RPC（行情推送、订单状态实时同步），比 HTTP 流式方案更高效；<br/>4. 同步 / 异步调用模型适配核心层事件驱动架构。 |
| 接入层（前端 / 第三方系统） ↔ API 网关 | HTTP     | 1. 前端 / 第三方系统对 HTTP 生态更熟悉，接入成本低；<br/>2. RESTful/OpenAPI 等标准化规范便于对接，API 网关的认证、限流等能力可无缝适配；<br/>3. API 网关可实现 “HTTP ↔ gRPC” 协议转换，兼顾外部易用性和内部高性能。 |
| 支撑服务层（微服务之间）               | gRPC     | 1. 微服务间多为 “操作语义” 调用（如回测引擎查历史数据），RPC 风格比 HTTP 更贴合；<br/>2. 二进制序列化 + HTTP/2 提升跨服务调用效率，适配时序数据库 / 回测引擎的批量数据交互；<br/>3. 强类型接口（Protobuf）减少微服务对接的序列化错误，符合量化交易严谨性要求。 |

协议分层架构，其中 API 网关负责协议转换、认证、限流，核心层专注低延迟处理，支撑层专注非实时性服务。

  ```plaintext
前端/第三方系统 → HTTP(RESTful) → API 网关 → gRPC → 支撑服务层 → gRPC → 核心交易层（C++20）
  ```



  ### 8.3 RESTful 使用场景规范（接入层 HTTP 接口专属）

  RESTful 是接入层 HTTP 接口的核心设计范式，需结合接口语义分场景适配：

  #### 8.3.1 适合 RESTful 的场景（接入层对外 HTTP 接口）

  **推荐设计为 RESTful**，核心理由：

  - 资源导向契合接入层操作：策略、订单、参数、账户等核心操作天然对应 “资源 CRUD”，RESTful 的 “URL+HTTP 方法”（GET/POST/PUT/DELETE）能清晰表达语义；

  - 标准化降低对接成本：RESTful 是行业通用规范，前端 / 第三方对接方无需额外学习成本，API 网关的认证、限流、路由可无缝适配其无状态特性；

  - 典型接口示例：

    - GET /api/v1/strategies/{id}：查询策略参数
    - PUT /api/v1/strategies/{id}/params：修改策略参数
    - POST /api/v1/orders：创建订单
    - DELETE /api/v1/orders/{id}：撤销订单

  #### 8.3.2 不适合 / 无需 RESTful 的场景

  - 核心层↔支撑层的 gRPC 接口：gRPC 基于 “服务 + 方法” 设计范式（如 `GetHistoricalData`/`RunBacktest`），强行适配 RESTful 会冗余且违背 RPC 设计初衷；
  - 流式交互接口（实时行情推送、订单状态流）：RESTful 基于 “请求 - 响应” 模型，无法高效支持持续流式传输，此类场景优先用 gRPC 流式 RPC（内部）或 WebSocket（接入层 HTTP）；
  - 复杂操作接口（策略回测、批量订单处理）：RESTful 的 HTTP 方法对 “多参数、多步骤” 复杂操作表达力不足，可设计为非 RESTful HTTP 接口（如 POST /api/v1/backtest，请求体传复杂参数），或通过 API 网关转 gRPC 调用内部服务。



## 九、企业级容灾与数据一致性设计

### 9.1 部署架构

- **生产环境**：采用 “双活机房 + 灾备机房” 部署模式，核心交易层在双活机房部署，支撑服务层跨机房集群化部署
- **核心交易层**：每个机房部署多实例，实例间通过本地 WAL 日志同步状态，跨机房实时同步关键数据（订单 / 持仓 / 资金）
- **支撑服务层**：采用多副本（3 副本）跨机房部署，数据通过分布式数据库 / 消息队列同步

### 9.2 故障切换策略

- **进程级故障**：核心进程崩溃后，监控系统触发自动重启，基于本地 WAL 日志恢复订单 / 持仓状态（RTO < 10 秒）
- **节点级故障**：单个节点故障后，流量自动切换至同机房备节点，核心数据通过共享存储 / 实时同步恢复（RTO < 30 秒）
- **机房级故障**：主机房故障后，灾备机房接管，通过跨机房同步数据恢复（RPO < 1 分钟，RTO < 5 分钟）

### 9.3 数据一致性保障

- **核心交易数据**：采用 WAL（预写日志）+ 内存快照机制，确保进程重启后数据不丢失；
- **跨机房数据同步**：采用 “实时增量同步 + 定时全量校验”，确保数据一致性；
- **订单幂等性**：通过全局唯一订单 ID 确保发单 / 撤单操作幂等，防止重复执行；
- **对账机制**：核心交易层与交易所 / 清算系统定时对账，发现数据不一致自动告警并触发人工核对。



## 十、企业级安全与合规设计

### 10.1 安全防护

- **网络安全**：核心交易层部署在私有网络，接入层通过防火墙 / 入侵检测系统（IDS）防护；API 网关支持 DDoS 防护、IP 白名单；
- **数据安全**：敏感数据（账户 / 资金 / 策略参数）加密存储（AES-256），传输加密（TLS 1.3）；数据访问需多因子认证；
- **代码安全**：策略 / 插件代码安全扫描，防止恶意代码；核心代码定期安全审计；
- **操作安全**：关键操作（如策略发布、资金划转）需双人复核；操作日志不可篡改，支持审计溯源。

### 10.2 合规适配

- **金融监管**：满足《证券期货经营机构私募资产管理业务管理办法》《期货交易管理条例》等监管要求，支持交易留痕、反洗钱、异常交易监控；
- **数据合规**：满足等保三级、数据安全法、个人信息保护法要求，支持数据脱敏、跨境数据合规；
- **审计合规**：审计日志留存≥3 年，支持司法取证；定期生成合规报告，对接监管报送系统。

此设计在保留原架构低延迟核心优势的基础上，补充了企业级系统所需的高可用、多租户、合规、可观测、容灾等能力，既满足量化交易对性能的极致要求，又适配金融机构对系统稳定性、合规性、可管理性的核心诉求，可支撑从中小机构到大型金融集团的量化交易业务需求。
