# 量化交易系统架构设计

## 一、系统定位与目标

### 1.1 系统定位

本系统定位为**面向机构客户的企业级多租户量化交易平台**，采用**纯行情驱动、单进程封闭交易引擎**的安全架构，支撑从策略研发、回测、仿真交易到实盘交易的全生命周期管理，适配高频交易场景，满足金融机构合规、高可用、高性能、可扩展、可审计的核心诉求，支持多账户、多交易所、多策略并行运行，具备跨地域部署与容灾能力。

### 1.2 核心目标

- **极致性能**：核心交易链路平均延迟 < 50 微秒，峰值订单处理能力 ≥ 10 万笔 / 秒

- **金融级高可用**：核心交易层全年可用性 ≥ 99.99%，支撑跨机房灾备，故障恢复时间（RTO）< 30 秒，数据恢复点目标（RPO）< 1 分钟

- **多租户隔离**：支持机构内多部门 / 多团队隔离，资源（CPU / 内存 / 行情 / 订单通道）按需分配，数据与操作权限严格隔离

- **合规可控**：满足金融监管要求（如交易留痕、反洗钱、异常交易监控），支持等保三级、数据脱敏与跨境数据合规

- **全生命周期可观测**：覆盖指标、日志、链路、调用链全维度监控，支持根因分析与性能溯源

- **弹性扩展**：核心层支持水平扩展，支撑服务层支持秒级弹性伸缩，插件化架构适配业务快速迭代

- **封闭安全**：交易策略**仅由内部行情事件驱动**，不提供任何外部控制接口，杜绝人为干预与非法操作风险

### 1.3 性能指标口径与热路径定义

为避免验收口径歧义，性能目标按下述**分段计量**，不在单一指标中混写「进程内处理」与「网络 / 跨服务」耗时。

- **「核心交易链路」**：指单笔业务在**核心进程内**的内存路径，典型分段为：标准化行情入队 → 策略回调与信号生成 → OMS 本地校验与写 WAL → 实时风控通过 → EMS 提交至适配器。**不包含**：跨进程网络往返（含与交易所 / 经纪商）、跨机房同步、支撑微服务同步调用、异步审计落盘的持久化延迟。

- **「平均延迟 < 50 微秒」**：适用于上述热路径在稳态、本地或专线接入条件下的**进程内处理分段目标**；与交易所 RTT、跨机房复制、支撑服务调用**分别验收**，对外承诺时需标明分段边界。

- **「峰值订单处理能力 ≥ 10 万笔 / 秒」**：指核心层多实例水平分片（如按品种 / 账户 / 租户路由）后的**聚合吞吐**能力；单进程 / 单实例极限需单独压测定义。

## 二、系统架构总览

### 架构层次：

本系统采用 “**核心模块进程内模块化集成 \+ 支撑功能微服务化 \+ 可插拔实现 \+ 企业级治理与容灾**” 的混合架构，平衡低延迟交易需求与企业级系统的稳定性、扩展性、合规性。

整体分为四层：

- **核心交易层**：进程内模块化设计，处理实时交易链路（微秒级响应），补充故障隔离、多核优化、本地容灾能力

- **支撑服务层**：独立部署的微服务集群，提供非实时性支撑功能，强化服务治理、熔断降级、弹性伸缩能力

- **接入层**：负责外部系统对接与用户交互，补充多租户、统一认证、安全网关能力

- **企业级治理层**：跨层治理能力，覆盖服务注册发现、配置灰度、容灾备份、合规审计、可观测性等企业级能力

### 核心设计原则：

- **低延迟优化**：核心交易链路全程内存交互，无网络开销，CPU 亲和性绑定，内存锁优化，网卡中断隔离

- **高可用性**：核心模块多实例部署 \+ 跨机房灾备，微服务集群化 \+ 熔断降级，关键组件主备切换 \+ 故障自动转移

- **可扩展性**：通过插件机制扩展数据源 / 策略 / 执行渠道，无需修改核心代码；支撑服务层支持水平扩容，核心层支持多进程分片部署

- **安全性**：多层风控（事前 / 事中 / 事后），操作审计，数据加密传输，多租户权限隔离，等保三级合规，敏感数据脱敏

- **可维护性**：模块化边界清晰，微服务独立升级，插件化降低耦合，全维度可观测，自动化运维（发布 / 回滚 / 监控）

- **多租户隔离**：资源隔离、数据隔离、权限隔离，租户级配置与策略独立管理

- **合规性**：全流程交易留痕，审计日志不可篡改，满足金融监管与行业合规要求

- **命名规范**：所有目录、文件、模块、接口严格遵循 **Google C\+\+ 命名规范**

- **负载均衡优化**：行情适配层采用多维度负载均衡策略，兼顾低延迟与负载均匀性，支持故障自动转移与动态调度

### 2.1 热路径与非热路径划分

- **热路径（交易核心）**：行情解析 → 事件总线 → 策略触发 → 实时风控 → OMS → EMS；**全程进程内内存交互**，**禁止任何同步阻塞、禁止调用支撑服务**。

- **非热路径（旁路异步）**：日志、监控、历史数据、备份、交易审计、回测；全部通过**grpc 异步请求 / 异步消息队列**旁路推送，不能影响交易延迟。

- **配置与风控**：采用**本地缓存 \+ 异步推送更新**，绝不阻塞交易热路径；配置更新仅通过消息队列异步下发，不主动调用交易引擎。

### 2.2 支撑服务分期建设建议

控制开局复杂度，按阶段扩展微服务边界：

- **一期（MVP）**：配置中心、日志采集、监控告警、服务发现；历史数据与回测简化部署，行情适配层基础负载均衡能力

- **二期**：策略研发平台、交易审计、容灾备份、报表体系，行情适配层动态负载均衡与灰度切换能力

- **三期**：跨机房灾备、监管报送自动化、智能风控，行情适配层跨机房负载均衡与全局调度能力

- **原则**：支撑服务与交易引擎**物理隔离**，可按需独立部署 / 关停，不影响实盘交易；行情负载均衡策略按需升级，不侵入核心交易链路

## 三、交易引擎层（单进程封闭运行，企业级优化）

所有模块运行在同一进程内，通过内存级事件总线通信，避免跨进程延迟。

封闭部署、无对外控制接口、仅通过适配器对接外部行情 / 交易所，策略完全由内部行情驱动。

|模块名称（Google C\+\+ 规范）|核心功能|可插拔设计|企业级优化点|
|---|---|---|---|
|**事件总线模块**|进程内事件中枢，模块间消息传递|核心基础设施，支持事件类型扩展|1. 无锁队列优化，按事件类型分片；2. 事件优先级（订单事件 > 行情事件 > 日志事件）；|
|**行情处理模块**|接收、清洗、标准化、分发行情数据|`IMarketSource` 适配器接口|1. 多数据源主备自动切换；2. 行情数据校验与异常过滤；3. 本地环形缓冲区缓存；4. 多核分片处理行情（按品种哈希分配 CPU）；5. 多维度行情负载均衡（品种分片 / 负载动态调整 / 优先级调度）|
|**策略引擎模块**|加载、运行、管理量化策略，生成交易信号|`IStrategy` 插件接口|1. 策略沙箱隔离，单个策略崩溃不影响核心进程；2. 策略资源限制（CPU / 内存 / 订单频率）；3. 策略热加载 / 卸载；4. 策略故障自动暂停并告警|
|**交易合规模块 (cms)**|实时交易合规校验（限仓、限购、日内频次）|规则可配置化|1. 多维度合规拦截；2. 合规日志本地落盘；3. 规则异步更新|
|**订单管理模块（oms）**|订单全生命周期管理（创建 / 修改 / 取消 / 状态跟踪）|核心流程固定，支持订单类型扩展|1. 订单本地 WAL 持久化，保证不丢单；2. 订单幂等性设计；3. 多租户订单隔离；4. 订单超时自动处理|
|**交易执行模块**|订单路由、拆单、算法执行、通道管理|`IExecutionAdapter` 适配器接口|1. 多通道自动切换与故障转移；2. 执行通道流量控制；3. 拆单算法支持；4. 滑点控制|
|**账号管理模块**|资金、资产、可用额度实时核算|无，进程内状态|1. 多租户账户隔离；2. 资金实时校验；3. 定时内存快照持久化|
|**订单管理模块**|持仓、开平、冻结、盈亏实时核算|无，进程内状态|1. 逐标的持仓管理；2. 今日 / 昨日持仓区分；3. 持仓异常自动校验|
|**风险管理模块**|事前 / 事中风控拦截（资金 / 持仓 / 亏损 / 涨跌停）|风险规则可配置化|1. 多维度风控（租户 / 账户 / 策略 / 品种）；2. 异常行情熔断；3. 风控阈值异步更新|

### 核心约束（强制）

1. 所有模块运行在**同一个交易引擎进程**内，不拆分、不独立部署

2. 交易引擎**不对外开放任何 TCP/HTTP 控制接口**，仅通过适配器对接外部行情与交易所

3. 策略**仅由内部行情 Tick/Bar 事件驱动**，不接受任何外部触发信号

4. 热路径上**禁止任何同步调用、禁止任何磁盘 I/O、禁止任何网络请求**

5. 行情负载均衡逻辑**必须在进程内完成**，禁止跨进程调用，确保微秒级调度延迟

## 四、支撑服务层（独立微服务，企业级治理）

所有服务**独立部署在非交易服务器**，通过`异步消息队列（Kafka）`与`交易引擎通信`，强化服务治理、弹性伸缩、容灾备份能力，满足企业级微服务架构要求。**不影响交易、不占用交易资源、可随意扩容 / 缩容 / 重启**。

|微服务名称|核心功能|技术特性|企业级优化点|
|---|---|---|---|
|配置中心服务|集中管理系统配置（数据源参数、策略参数、风控阈值等）|支持动态配置更新，配置版本管理，权限控制|1. 配置灰度发布（按租户 / 模块 / 实例维度）；2. 配置变更审计日志（谁改 / 改了什么 / 何时改）；3. 配置回滚能力，支持配置生效状态校验；4. 多租户配置隔离，租户仅可见自身配置；5. 行情负载均衡策略配置化管理（分片规则 / 阈值 / 优先级）|
|历史数据服务|存储 / 查询历史行情、成交记录、策略回测数据|基于时序数据库（如 ClickHouse），支持高效批量查询，数据自动归档|1. 数据多副本存储（3 副本），跨机房同步；2. 数据生命周期管理（热数据 / 冷数据分层存储，冷数据归档至对象存储）；3. 数据访问权限控制（按租户 / 品种 / 时间范围）；4. 数据脱敏（隐藏敏感账户信息）；5. 批量导出合规化（带水印 / 审计日志）|
|日志分析服务|收集 / 存储 / 检索系统日志与交易流水|支持日志分级、按模块过滤，提供全文检索，日志可视化分析|1. 日志不可篡改（写入后追加模式，哈希校验）；2. 日志分级存储（核心交易日志独立存储，保存≥3 年）；3. 多租户日志隔离，支持按租户检索；4. 日志与调用链关联，支持根因分析；5. 行情负载均衡切换日志独立存储、审计可追溯|
|业务监控服务|业务指标（策略收益 / 订单量）、行情适配层负载指标（CPU / 队列长度 / 切换次数）|实时指标采集，自定义告警阈值，多渠道通知（邮件 / 短信 / 钉钉）|1. 全维度可观测（指标 \+ 日志 \+ 链路 \+ 进程内埋点）；2. 告警分级（P0\-P4），支持告警升级规则；3. 租户级监控视图，机构级汇总视图；4. 性能基线自动生成，异常检测（同比 / 环比）；5. 监控数据长期存储（≥1 年），支持趋势分析；6. 行情负载均衡指标实时监控（分片负载、切换频率、延迟波动）|
|回测引擎服务|基于历史数据验证策略有效性|支持多线程并行回测，模拟滑点 / 手续费，生成绩效报告（夏普比率 / 最大回撤）|1. 回测资源弹性伸缩（基于队列自动扩缩容）；2. 回测结果校验（与实盘数据对比）；3. 回测参数版本管理，支持回测结果复现；4. 多租户回测资源隔离，防止资源抢占；5. 回测报告合规化（包含风险提示、数据来源说明）|
|策略研发服务|策略代码管理、编译、测试|集成 Git 版本控制，自动化编译插件，策略安全审计|1. 策略代码安全扫描（防恶意代码 / 违规调用）；2. 策略编译沙箱，防止编译过程影响系统；3. 策略版本灰度发布，支持灰度放量；4. 策略研发权限控制（读 / 写 / 编译 / 发布分级）；5. 策略代码审计日志，满足合规要求|
|交易审计服务|盘后风险分析与合规审计|生成每日 / 每月风控报告，检查是否符合监管要求，异常交易行为分析|1. 监管合规报表自动生成（如反洗钱、异常交易报告）；2. 盘后批量复盘当日所有订单、成交、持仓；3. 统计超限行为、异常交易记录；4. 违规行为日志归档、审计追溯；5. 行情数据源切换合规性校验（是否符合容灾策略）|
|服务注册发现服务|微服务注册、发现、健康检查|基于 etcd 实现，支持服务健康状态实时检测|1. 服务熔断降级，防止雪崩；2. 服务负载均衡（按实例性能 / 负载）；3. 服务下线灰度，支持优雅退出；4. 跨机房服务发现，优先调用同机房实例|
|容灾备份服务|核心数据备份、跨机房同步、故障恢复|基于增量备份 \+ 实时同步，支持一键恢复|1. 核心交易层数据实时同步至备机房；2. 定时全量备份（每日）\+ 增量备份（分钟级）；3. 备机房热备，故障自动切换；4. 恢复流程自动化，支持 RTO/RPO 达标验证；5. 行情适配层配置与负载均衡策略同步备份，故障恢复后自动恢复|

### 核心约束（强制）

1. 所有支撑服务**不得与交易引擎部署在同一物理机 / 虚拟机**

2. 支撑服务**禁止主动调用交易引擎**，仅允许 配置中心服务通过消息队列异步推送配置更新

3. 交易引擎向支撑服务的所有通信**均为异步单向推送**，不等待响应

4. 行情负载均衡策略配置更新仅能通过配置中心异步推送，禁止直接修改交易引擎内存配置

## 五、接入层（企业级安全与多租户优化）

接入层**仅提供数据查询、监控查看、配置管理（推送到配置中心）、回测提交、审计报表展示**功能，**完全无法操作交易引擎、无法下单、无法启停策略、无法修改交易状态**。

|组件名称|功能说明|部署方式|企业级优化点|
|---|---|---|---|
|前端控制台|可视化操作界面（策略启停、参数调整、订单监控、绩效展示）、行情适配层负载监控 / 策略配置|Web 应用（独立部署，通过 API 网关与后端通信）|1. 多租户视图隔离，租户仅可见自身数据；2. 操作权限细粒度控制（按钮级 / 功能级）；3. 前端操作日志全记录，支持操作溯源；4. 敏感数据前端脱敏（如账户号隐藏中间位）；5. 支持多角色（管理员 / 交易员 / 审计员）；6. 行情负载均衡策略可视化配置（分片规则 / 优先级 / 阈值）|
|API 网关|统一接口入口，处理认证、限流、请求路由|独立部署，作为微服务与前端 / 外部系统的交互中枢|1. 统一身份认证（支持 OAuth2.0/SAML/ 企业 SSO）；2. 租户级限流（QPS / 并发数）；3. API 调用审计日志，不可篡改；4. API 版本管理，支持灰度发布；5. 防重放 / 防篡改 / 防 SQL 注入等安全防护；6. 跨域访问控制，符合 CORS 规范|
|第三方系统适配器|对接外部系统（如资管平台、清算系统）|独立部署，按需扩展适配器类型|1. 适配器接入认证，防止未授权对接；2. 数据传输加密（TLS 1.3）；3. 对接日志全记录，支持对账；4. 适配器故障隔离，单个适配器故障不影响整体；5. 对接数据格式标准化，支持多版本协议兼容|
|多租户管理组件|租户创建、资源分配、权限配置|集成至 API 网关与配置中心，独立部署|1. 租户生命周期管理（创建 / 暂停 / 注销）；2. 资源配额管理（CPU / 内存 / 行情 / 订单数）；3. 租户级数据隔离（数据库分库 / 分表）；4. 租户权限继承与自定义，支持角色模板；5. 租户级行情负载均衡策略隔离（不同租户独立分片规则）|

### 核心约束（强制）

1. 接入层**不提供任何直接操作交易引擎的接口**

2. 所有配置修改必须通过 config_service 异步下发，不能直接修改交易引擎内存

3. 前端控制台**不提供所有交易操作按钮**，仅保留查询与配置提交功能

4. 行情负载均衡策略配置仅能通过接入层提交至配置中心，禁止直接下发至交易引擎

## 七、适配层设计详细说明

### 7.1 行情数据源适配器

#### 7.1.1 整体架构新增负载均衡核心组件

在原有适配器架构基础上，新增**行情负载均衡器（MarketLoadBalancer）** 核心组件，与数据源管理器深度集成，负责行情数据接收、分片、调度、故障转移全流程，确保行情处理的低延迟、高可用与负载均匀性。

#### 7.1.2 抽象接口层（新增负载均衡扩展接口）

定义`IMarketSource`标准接口，包含：

- 连接管理：`Connect(config)`、`Disconnect()`

- 订阅控制：`Subscribe(instruments)`、`Unsubscribe(instruments)`

- 数据回调：`OnTick(tick_data)`、`OnBar(bar_data)`

- 状态查询：`IsConnected()`、`GetSupportedInstruments()`

- 负载监控扩展：`GetLoadMetrics()`（返回 CPU 使用率 / 队列长度 / 处理延迟）、`SetShardingRule()`（设置品种分片规则）

- 企业级扩展：`GetHealthStatus()`（健康状态上报）、`GetErrorCode()`（标准化错误码）、`SetPriority()`（数据源优先级）、`GetShardingId()`（获取当前分片 ID）

#### 7.1.3 负载均衡核心策略

行情负载均衡器实现以下核心策略，所有逻辑均在进程内完成，确保微秒级调度延迟：

|负载均衡策略|核心逻辑|技术实现|企业级优化点|
|---|---|---|---|
|**品种哈希分片**|按品种 ID 哈希映射到固定分片（CPU 核 / 适配器实例），分片数与 CPU 核数匹配|一致性哈希算法，预计算品种 \- 分片映射表，本地缓存|1. 分片映射表异步更新，不阻塞行情处理；2. 分片迁移时平滑过渡，无数据丢失|
|**动态负载调整**|实时监控各分片负载（CPU>80%/ 队列长度 > 1000 / 处理延迟 > 20μs 触发调整）|滑动窗口统计负载指标，超过阈值时迁移低优先级品种至空闲分片|1. 负载阈值可配置；2. 仅迁移非高频品种，避免影响核心交易；3. 调整日志实时上报|
|**优先级调度**|主数据源（优先级 1）优先处理，备数据源（优先级 2\+）热备，故障时毫秒级切换|心跳检测（10ms 间隔）\+ 本地行情缓存，切换时基于 WAL 恢复未处理行情|1. 切换前灰度验证备数据源可用性；2. 切换延迟 < 5ms；3. 切换记录全量审计|
|**行情类型分流**|Tick 数据（高优先级）与 Bar 数据（低优先级）分流至不同分片处理|事件总线按类型分片，独立无锁队列|1. Tick 分片绑定独占 CPU 核；2. Bar 分片复用空闲资源，不抢占 Tick 处理资源|
|**流量控制限流**|按数据源限流规则（如每秒最大 Tick 数）分配流量，防止单数据源过载|令牌桶算法，动态调整令牌生成速率，超限行情本地缓存（缓存深度可配置）|1. 限流阈值按租户 / 品种分级；2. 缓存溢出告警，避免数据丢失|
|**多核负载均衡**|单进程内按 CPU 核分片，绑定 CPU 亲和性，禁止跨核调度|CPU 亲和性绑定（sched_setaffinity），中断隔离|1. 核间无共享内存；2. 避免 CPU 上下文切换，降低延迟波动|

#### 7.1.4 插件实现层（负载均衡适配）

- 每种数据源（如 CTP、Binance、Wind）实现`IMarketSource`接口，兼容负载均衡扩展接口

- 编译为独立动态库（.so/.dll），包含插件元数据（支持的品种、数据类型、版本、兼容系统版本、支持的负载均衡策略）

- 实现数据协议解析与转换（统一为系统内部格式）

- 企业级扩展：1. 插件签名校验，防止恶意插件；2. 插件版本兼容检测，避免版本冲突；3. 插件运行状态监控，上报资源占用 / 错误率；4. 插件支持按分片 ID 过滤行情，仅处理分配的品种

#### 7.1.5 管理组件（集成负载均衡能力）

- 数据源管理器：负责插件加载、配置解析、生命周期管理，集成行情负载均衡器核心逻辑

- 数据融合器：多数据源时按优先级合并数据（如主备数据源切换），按分片规则分发至对应处理单元

- 插件注册中心：维护可用数据源插件列表与版本信息，记录各插件分片负载状态

- 负载均衡监控器：实时采集各分片 / 插件的负载指标，触发动态调整或告警

- 企业级扩展：

    1. 插件沙箱，隔离插件运行环境；

    2. 插件灰度加载，支持部分实例升级；

    3. 插件故障自动卸载，降级至备用插件；

    4. 插件操作审计，记录加载 / 卸载 / 切换日志；

    5. 负载均衡策略热更新（无需重启引擎）；

    6. 跨机房行情源负载均衡（三期扩展，按机房网络延迟动态调度）；

    7. 租户级负载隔离（不同租户行情分片独立，避免资源抢占）

### 7.2 交易执行模块适配器

- **抽象接口层**：定义`IExecutionAdapter`接口，包含：

    - 连接管理：`Connect(config)`、`Disconnect()`

- **适配器实现层**：

### 7.3 策略插件（Strategy Plugin）

- **抽象基类层**：定义`IStrategy`基类，包含：

## 八、模块与微服务交互方式（企业级可靠性增强）

### 8.1 交互方式说明

- **交易引擎内部**：内存事件总线（无锁队列），同步 / 异步事件驱动；行情负载均衡器与各适配器 / 分片间通过内存队列通信，无网络开销

- **交易引擎 → 支撑服务**：**仅异步消息队列（Kafka）**，绝不同步调用，不等待响应；行情负载均衡指标 / 切换日志异步推送至日志 / 监控服务

- **支撑服务 → 交易引擎**：**禁止主动调用**，仅允许 config_service 通过 Kafka 异步推送配置更新（含行情负载均衡策略配置）

- **外部系统 → 交易引擎**：**完全禁止**，所有外部请求只能通过接入层访问支撑服务

### 8.2 核心数据流（最关键）

行情触发策略的数据流（新增负载均衡环节）：

```Plain
外部行情源 → 行情适配器 → 行情负载均衡器（品种分片/动态调度）→ MarketHandler → EventBus（按分片隔离）→ StrategyEngine 
→ RiskManager → Oms → Ems → 执行适配器 → 交易所/券商柜台
```

交易引擎发送异步消息到 service 层的数据流：

```Plain
交易引擎 →（异步单向推送）→ Kafka → 支撑服务（log/monitor/history/backup/audit）
```

配置更新的数据流（新增行情负载均衡策略）：

```Plain
外部用户 → API 网关 → 配置中心服务（行情负载均衡策略校验）→ Kafka → 交易引擎 → 行情负载均衡器（异步更新策略）
```

外部 API 请求的数据流：

```Plain
外部用户 → API 网关 → 支撑服务（查询/配置提交/回测/行情负载均衡指标查询）
```

### 8.3 协议规范

|交互场景|推荐协议|核心理由|
|---|---|---|
|交易引擎内部|内存结构体 \+ 无锁队列|同进程内部调用，行情负载均衡调度需微秒级响应|
|交易引擎 ↔ 适配器|函数调用 \+ 回调接口|同进程内部的事件回调，行情负载均衡器直接调用适配器接口|
|交易引擎 ↔ 支撑服务|Protobuf \+ Kafka|1. Protobuf 二进制序列化比 HTTP\+JSON 快数倍，HTTP/2 多路复用减少连接开销，降低毫秒级延迟；<br>2. C\+\+20 对 gRPC 生态支持完善（协程、强类型），契合核心层开发栈；<br>3. 原生支持流式 RPC（行情推送、订单状态实时同步），比 HTTP 流式方案更高效；<br>4. 同步 / 异步调用模型适配核心层事件驱动架构；<br>5. 行情负载均衡指标二进制序列化，减少传输开销|
|支撑服务之间|gRPC \+ Protobuf|1. 微服务间多为 “操作语义” 调用（如回测引擎查历史数据），RPC 风格比 HTTP 更贴合；<br>2. 二进制序列化 \+ HTTP/2 提升跨服务调用效率，适配时序数据库 / 回测引擎的批量数据交互；<br>3. 强类型接口（Protobuf）减少微服务对接的序列化错误，符合量化交易严谨性要求|
|接入层 ↔ 外部系统|HTTP/HTTPS \+ RESTful|1. 前端 / 第三方系统对 HTTP 生态更熟悉，接入成本低；<br>2. RESTful/OpenAPI 等标准化规范便于对接，API 网关的认证、限流等能力可无缝适配；<br>3. API 网关可实现 “HTTP ↔ gRPC” 协议转换，兼顾外部易用性和内部高性能；<br>4. 行情负载均衡策略配置通过 RESTful API 提交，适配前端可视化配置|

## 九、企业级容灾与数据一致性设计

### 9.1 部署架构

- **生产环境**：采用 “双活机房 \+ 灾备机房” 部署模式，核心交易层在双活机房部署，支撑服务层跨机房集群化部署

- **核心交易层**：每个机房部署多实例，按品种 / 租户 / 策略组分片；实例间独立运行，无共享状态；行情负载均衡器支持跨机房数据源调度（三期）

- **支撑服务层**：采用多副本（3 副本）跨机房部署，数据通过分布式数据库 / 消息队列同步

- **网络隔离**：交易引擎部署在独立内网，与支撑服务、接入层通过防火墙隔离

- **行情适配层部署**：每个核心交易实例内置行情负载均衡器，多实例间按品种分片，避免单实例过载；主备数据源跨机房部署，提升容灾能力

### 9.2 故障切换策略

- **进程级故障**：核心进程崩溃后，监控系统触发自动重启，基于本地 WAL 日志恢复订单 / 持仓状态（RTO < 10 秒）；行情负载均衡策略配置从本地缓存恢复，无需依赖外部服务

- **节点级故障**：单个节点故障后，流量自动切换至同机房备节点，核心数据通过共享存储 / 实时同步恢复（RTO < 30 秒）；行情分片自动迁移至备节点，迁移延迟 < 50ms

- **机房级故障**：主机房故障后，灾备机房接管，通过跨机房同步数据恢复（RPO < 1 分钟，RTO < 5 分钟）；行情数据源自动切换至灾备机房接入点，负载均衡策略同步切换

- **行情源故障**：行情负载均衡器检测到数据源心跳异常后，毫秒级切换至备数据源，基于本地缓存兜底，无数据丢失

### 9.3 数据一致性保障

- **核心交易数据**：采用 WAL（预写日志）\+ 内存快照机制，确保进程重启后数据不丢失；

- **跨机房数据同步**：采用 “实时增量同步 \+ 定时全量校验”，确保数据一致性；

- **订单幂等性**：通过全局唯一订单 ID 确保发单 / 撤单操作幂等，防止重复执行；

- **对账机制**：核心交易层与交易所 / 清算系统定时对账，发现数据不一致自动告警并触发人工核对；

- **行情数据一致性**：行情负载均衡器对主备数据源的数据进行校验，确保切换后数据无重复 / 丢失；分片迁移时通过 WAL 日志同步未处理行情，保证数据一致性

## 十、企业级安全与合规设计

### 10.1 安全防护

- **网络安全**：核心交易层部署在私有网络，接入层通过防火墙 / 入侵检测系统（IDS）防护；API 网关支持 DDoS 防护、IP 白名单；

- **数据安全**：敏感数据（账户 / 资金 / 策略参数）加密存储（AES\-256），传输加密（TLS 1.3）；数据访问需多因子认证；行情负载均衡策略配置加密存储，仅授权角色可修改

- **代码安全**：策略 / 插件代码安全扫描，防止恶意代码；核心代码定期安全审计；行情负载均衡核心逻辑代码独立审计，防止逻辑漏洞

- **操作安全**：关键操作（如策略发布、资金划转、行情负载均衡策略修改）需双人复核；操作日志不可篡改，支持审计溯源

### 10.2 合规适配

- **金融监管**：满足《证券期货经营机构私募资产管理业务管理办法》《期货交易管理条例》等监管要求，支持交易留痕、反洗钱、异常交易监控；行情数据源切换记录纳入合规审计，满足监管留痕要求

- **数据合规**：满足等保三级、数据安全法、个人信息保护法要求，支持数据脱敏、跨境数据合规；

- **审计合规**：审计日志留存≥3 年，支持司法取证；定期生成合规报告，对接监管报送系统；行情负载均衡操作日志独立归档，支持监管溯源

## 十一、核心风险点与架构应对措施

|风险点|架构层面应对措施|新增行情负载均衡相关应对|
|---|---|---|
|行情源中断|多数据源主备自动切换；本地行情缓存兜底|行情负载均衡器毫秒级切换至备数据源，按分片隔离故障|
|行情处理节点过载|\-|动态负载调整，迁移低优先级品种至空闲分片；多核分片隔离|
|交易通道故障|多通道自动切换；订单本地缓存，故障恢复后自动重发|\-|
|策略异常|策略沙箱隔离；资源限制；异常自动暂停；风控熔断|策略绑定指定行情分片，故障仅影响单分片策略|
|交易引擎崩溃|本地 WAL 日志 \+ 内存快照，秒级重启恢复|行情负载均衡策略本地缓存，重启后自动恢复|
|配置更新异常|配置灰度发布；版本回滚；本地缓存兜底|行情负载均衡策略灰度更新，异常时自动回滚|

|人为干预|交易引擎封闭运行；接入层仅查询权限；操作审计|行情负载均衡策略修改需双人复核，操作日志不可篡改|
