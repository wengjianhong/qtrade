# 开发指南

本文档说明**仓库目录约定与研发协作要点**。系统设计目标、模块边界与交互规则以 [`Architecture.md`](Architecture.md) 为权威来源；实现演进时需与该文档同步评审。


## 1. 与架构文档的对应关系

| 《Architecture.md》层次 | 代码侧落到何处 |
| ----------------------- | -------------- |
| 核心交易层 | `src/core/`（进程内：事件总线、行情、策略、OMS、EMS、账户、实时风控） |
| 支撑服务层 | `src/service/<名称>/`（独立进程 / 镜像部署；与核心通过 gRPC / 消息队列交互） |
| 接入层 | `api/gateway/`、`api/console/`（可按里程碑增补目录） |
| 企业级治理 | 多与接入层、运维流水线耦合：`release/deploy/`、`release/ci/`、密钥与策略以外置配置为主 |

**性能口径**：核心进程内热路径定义、与非热路径边界见《架构》**§1.3、§2.1、§8.4**；开发时禁止在策略回调同线程路径上同步阻塞等待远程服务。


## 2. 代码结构（目标布局与当前仓库）

仓库根目录名可与克隆方式一致（如 `quant-trading/`）；下列树状结构与《架构》**第三节～第七节**及 **`Guide` 历史约定**对齐，并兼容当前已存在的 `src/core`、`src/plugins`、`src/service` 布局。

```shell
quant-trading/
├── docs/
│   ├── Architecture.md        # 系统架构权威文档
│   └── Guide.md               # 本开发指南
├── include/                   # 公共头文件（数据结构、跨模块接口等）
├── src/
│   ├── core/                  # 核心交易层（进程内模块）
│   │   ├── event_bus/         # 事件总线
│   │   ├── market_data/       # 行情（IDataSource 插件接口）
│   │   ├── strategy/          # 策略引擎（StrategyBase 插件）
│   │   ├── oms/               # 订单管理
│   │   ├── ems/               # 执行管理（IExecutionAdapter 插件）
│   │   ├── account/           # 账户与持仓
│   │   └── risk/              # 实时风控
│   ├── plugins/               # 可插拔动态库（.so / .dll）
│   │   ├── data_sources/      # 
│   │   ├── strategies/        # 
│   │   └── executors/         # 
│   └── service/               # 支撑微服务（各子目录可独立构建与部署）
│       ├── config_service/    # 
│       ├── history_data/      # 历史数据（按需）
│       ├── logging_service/   # 
│       ├── monitoring/        # 
│       ├── backtest/          # 
│       ├── strategy_dev/      # 策略研发流水线（二期可建）
│       ├── risk_audit/        # 事后风控审计（二期可建）
│       ├── registry/          # 服务注册发现客户端或适配（与 etcd 等对接）
│       └── disaster_recovery/ # 备份编排 / 恢复自动化（可与运维平台合并）
├── api/                       # 接入层（按里程碑建立）
│   ├── gateway/               # API 网关
│   └── console/               # 前端控制台
├── test/                      # 单元 / 集成测试
└── release/                   # 发行与运维
    ├── ci/
    ├── deploy/
    └── docs/
```

**说明**：尚未创建的目录（如 `api/`、部分 `service/*`）可在对应里程碑落地时补齐；《架构》**§2.2** 允许一期先合并「可观测 + 日志管道」等边界，避免过早拆分空目录。

若日后将 `base/`（通用线程池、无锁结构等）从 `include/` + `src/` 中独立出来，保持与《架构》「共享基础代码」职责一致即可。


## 3. 热路径与非热路径（开发约束摘要）

- **热路径**：行情 → 策略 → OMS → 实时风控 → EMS；以内存与本地 WAL 为主。**不得**在此路径上同步 gRPC 等待配置中心 / 历史库等；配置采用本地缓存 + 推送失效（详见《架构》§2.1、§8.4）。
- **非热路径**：回测、报表、日志检索、事后审计、批量查询等，可使用 gRPC、MQ，毫秒～秒级延迟可接受。
- **背压**：当产出快于 EMS / 交易所吞吐时，由 OMS / 事件总线按策略限流、排队上限或暂停策略，避免无界内存增长（《架构》§11.2）。


## 4. 协议与集成要点

- **核心 ↔ 支撑层**：优先 gRPC（Protobuf）；热路径禁止依赖同步 RPC。
- **接入层 ↔ 外部**：HTTP(S) / WebSocket；网关可做 HTTP ↔ gRPC 转换（《架构》§8.2、§8.3）。
- **消息队列**：至少一次投递时，消费侧必须 **幂等**（业务键去重）；顺序敏感主题按租户 / 账户 / 品种分区。


## 5. 插件、Protobuf 与版本

- **动态插件**：`IDataSource` / `StrategyBase` / `IExecutionAdapter` 实现编译为独立 `.so`，加载前签名校验；**ABI** 与核心二进制需在发布说明中锁定兼容矩阵（《架构》§11.3）。
- **接口演进**：`.proto` 以向后兼容方式演进；破坏性变更通过新 service / package 版本并行发布。


## 6. 关键设计细节

- **共享基础代码**：统一 Tick / Order 等类型放在 `include/`（或后续 `base/`），避免跨模块重复定义。
- **插件独立编译**：`plugins/` 与核心分离链接，仓库内联调、运行时装载，边界与《架构》第七章一致。
- **核心与微服务同仓**：接口变更时可同时改 `src/core` 与 `src/service/*`，仍保持**部署独立**、**热路径不阻塞**。
- **可观测性**：指标与 Trace 在热路径采用低开销埋点 + 异步导出 + 采样，避免破坏微秒级预算（《架构》§11.4）。


## 7. 仓库演进与多仓拆分

当微服务数量或团队边界扩大时：

- 优先将**成熟、接口稳定**的服务（如日志、监控采集）拆至独立仓库；
- **核心交易层**与强依赖服务（如历史数据）可留在主仓，缩短迭代路径；
- 依赖通过 **git submodule**、**CMake FetchContent** 或内部制品库管理；
- 拆分前后持续满足《架构》**§9.4**（容灾、脑裂与一致性表述）与 **§11.6**（回放与混沌演练）的验证要求。
